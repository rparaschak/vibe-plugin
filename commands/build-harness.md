## Doctor mode (a re-run is the diagnostic — not a separate branch)
The pipeline is the same on first build and re-run; doctor behavior is emergent from state, not a runtime branch:
- **Re-audit** — the audit step marks each checklist row PRESENT or ABSENT against the live `HARNESS_ROOT/`. Command rows use the **prefix-aware** target path for the resolved `COMMAND_PREFIX`. If `COMMAND_PREFIX` is non-empty and an old flat `HARNESS_ROOT/commands/<cmd>.md` exists while the namespaced target is missing, mark the command row `not_started` (or note the gap) — re-stamp into the namespaced path; **do not** silently delete the flat file (surface it for the user).
- **Upgrade, edit-safe** — a stamped section that is unmodified but drifted from its current template is re-stamped by a stamp worker (rewriting only its `generated` date); anything edited below the marker is surfaced for the user, never overwritten. **Frontmatter exception:** agent frontmatter `model:` and `effort:` values are project-tunable overrides — a re-stamp preserves the existing stamped file's values even when they differ from the template's defaults; they are never drift. All other agent frontmatter (`name`, `description`, `skills`, `color`) remains template-governed.
- **Legacy supersession** — same-role twins (`be-review` beside `backend-review`; a pre-rename `vibe-<kernel-name>` skill dir like `vibe-research-protocol` beside its unprefixed successor `research-protocol`) are decided at the checklist gate, never silently: **adopt & retire** removes the legacy file only after its content is folded into the stamped successor and nothing retained still references it; **retain** keeps it side-by-side. Doctor re-collects twins on every run — a retained twin resurfaces as a disposition at the next re-run's gate. Finalize reports the **retirement list**: each remaining legacy file → its stamped successor → what still references it. No deletion ever happens outside a gate-approved disposition.
- **Idempotent** — on an unchanged, up-to-date project the build step finds every row already `passing`, so it re-emits nothing. `.workspace/harness/checklist.md` itself is re-written only when its content differs, and its Handoff block carries no run timestamp.
- **Fresh-session test** — after re-audit, verify a cold scout could answer *what is this / how to run / how to verify / what's unfinished* from repo contents alone; a gap becomes a new `not_started` row.
- **Coherence audit (whole-harness — after all stamps, before Finalize)** — one read-only scout sweeps the assembled `HARNESS_ROOT` and reports four checks: (a) **reference closure** — every skill/agent/command name referenced by any harness file resolves to an existing file (a dangling legacy name like `be-review` in a command is a finding); (b) **single authority** — no two skills cover the same role or domain token without a gate disposition recorded; (c) **slot closure** — zero literal `{{…}}` anywhere under `HARNESS_ROOT` (grep, mechanical); (d) **stack fit** — no stamped file carries a code fence in a language the repo's markers don't justify. Every finding becomes a `blocked` or `not_started` row with the file cited. Per-row audits prove each stamp; only this audit proves the harness.
